# AGENTS.md

## Codex Operating Mode: Preview-First Delivery

This repository uses a **preview-first** workflow.

Codex must treat every implementation task as incomplete until the work is committed, pushed, pull-requested, and preview-deployed, unless preview deployment is technically impossible.

Do not interpret requests such as "remove", "delete", "disable", "hide", "turn off", or "clean up" as permission to skip deployment. Those are still code changes and must go through the full preview-first workflow.

---

## Non-Negotiable Rules

1. **Never finish with uncommitted changes.**
2. **Never work directly on `main`, `master`, `production`, or a protected production branch** unless explicitly instructed.
3. **Never deploy an individual task branch directly to production.**
4. **Every task branch must have its own preview deployment** when preview deployment is configured for the repo.
5. **Parallel tasks must stay isolated** in separate branches or worktrees.
6. **Overlapping parallel tasks must be integrated through an integration branch.**
7. **Do not overwrite, reset, or force-push another task branch.**
8. **Do not invent a deployment URL.** If no preview URL is available, report the exact blocker.

---

## Definition of Done

A task is complete only when all applicable items below are done:

1. A dedicated branch exists for the task.
2. The requested change is implemented.
3. Relevant validation commands have been run.
4. All changes are committed.
5. The branch is pushed.
6. A pull request is created or updated.
7. A preview deployment exists for the branch, or a precise deployment blocker is reported.
8. The final response includes:
   - branch name
   - commit hash
   - PR URL
   - preview deployment URL, or exact blocker if unavailable
   - validation commands run
   - validation result
   - files changed
   - known risks or unresolved issues

## Branching Rules

Create a dedicated branch for every task.

Branch format:

```text
codex/<short-task-name>
```

Examples:

```text
codex/remove-legacy-banner
codex/fix-checkout-total
codex/add-user-preview-card
codex/disable-unused-webhook
```

Before creating a branch:

1. Check the current branch.
2. Check for uncommitted changes.
3. Do not discard user changes.
4. If unrelated uncommitted changes exist, stop and report them clearly.

---

## Commit Rules

Every task must produce at least one commit.

Commit message format:

```text
<type>: <clear task summary>
```

Allowed types:

```text
feat
fix
chore
refactor
test
docs
style
perf
ci
build
```

Examples:

```text
fix: disable legacy checkout banner
feat: add order preview page
chore: remove unused analytics script
refactor: isolate payment summary rendering
```

Do not leave generated files, temporary files, debug logs, or local secrets in the commit.

## Validation Rules

Run the strongest relevant validation available in the repository.

First inspect the repo to determine the stack and available scripts:

- `package.json`
- `pnpm-lock.yaml`
- `yarn.lock`
- `package-lock.json`
- `bun.lockb`
- `requirements.txt`
- `pyproject.toml`
- `Pipfile`
- `poetry.lock`
- `go.mod`
- `Cargo.toml`
- `Gemfile`
- CI config files
- existing README or developer docs

For JavaScript or TypeScript projects, prefer available scripts in this order when present:

```text
lint
typecheck
test
build
```

Use the repository's package manager:

```text
pnpm if pnpm-lock.yaml exists
yarn if yarn.lock exists
npm if package-lock.json exists
bun if bun.lockb exists
```

Examples:

```text
pnpm lint
pnpm typecheck
pnpm test
pnpm build
```

```text
npm run lint
npm run typecheck
npm test
npm run build
```

For other stacks, use the repo's documented validation commands.

If a validation command fails:

1. Fix the issue if it is caused by the task.
2. Re-run validation.
3. If the failure is pre-existing or unrelated, report evidence clearly.
4. Do not hide failed validation.

## Preview Deployment Rules

Every task must be preview-deployed when the repository has a preview deployment mechanism.

Codex must inspect the repository and connected workflow to determine the preview deployment path. Look for:

- Vercel configuration
- Netlify configuration
- Cloudflare Pages configuration
- GitHub Actions deployment workflows
- Render configuration
- Railway configuration
- Fly.io configuration
- Docker deployment docs
- README deployment instructions
- CI/CD comments or checks on the pull request

### Vercel

If the repo uses Vercel Git integration:

1. Push the task branch.
2. Create or update the PR.
3. Use the Vercel PR preview deployment generated by the Git integration.
4. Report the preview URL from the PR checks/comments when available.

If the Vercel CLI is configured and authenticated, a non-production preview deploy may be used.

Do not run production deployment commands such as:

```text
vercel --prod
```

unless explicitly instructed after merge to the production branch.

### Netlify

If the repo uses Netlify deploy previews:

1. Push the task branch.
2. Create or update the PR.
3. Use the Netlify deploy preview generated by the Git integration.
4. Report the preview URL from PR checks/comments when available.

If the Netlify CLI is configured and authenticated, a non-production deploy may be used.

Do not deploy production unless explicitly instructed after merge to the production branch.

### Cloudflare Pages

If the repo uses Cloudflare Pages preview deployments:

1. Push the task branch.
2. Create or update the PR.
3. Use the Cloudflare Pages preview generated by the Git integration.
4. Report the preview URL from PR checks/comments when available.

Do not deploy production unless explicitly instructed after merge to the production branch.

### Unknown or Missing Preview Deployment

If no preview deployment mechanism is available:

1. Still create the branch.
2. Still implement the task.
3. Still run validation.
4. Still commit.
5. Still push.
6. Still create or update the PR.
7. Report that preview deployment is blocked.
8. State the exact missing requirement.

Do not say only:

```text
not deployed
```

Instead report:

```text
Preview deployment blocked because <exact reason>.
Required next step: <specific configuration or credential needed>.
```

## Parallel Task Rules

This repository may have multiple Codex tasks running in parallel.

For every parallel task:

1. Use a separate branch or worktree.
2. Do not modify another task branch.
3. Do not reset another task branch.
4. Do not force-push another task branch.
5. Do not assume another task should lose.
6. Preserve the intent of other task branches when integrating.
7. Each task branch must have its own PR and preview deployment when available.

Parallel work should not be stacked accidentally.

Bad pattern:

```text
task-a branch includes task-b changes by accident
task-b branch overwrites task-a changes
last deployed branch silently wins
```

Correct pattern:

```text
main
├── codex/task-a              -> PR A -> preview A
├── codex/task-b              -> PR B -> preview B
└── codex/integration-a-b     -> PR Integration -> preview Integration
```

---

## Integration Branch Rules

If two or more parallel tasks overlap, interact, or modify related behavior, create an integration branch.

Integration branch format:

```text
codex/integration-<task-a>-<task-b>
```

Example:

```text
codex/integration-checkout-banner-payment-summary
```

Integration workflow:

1. Start from the latest target branch, usually `main`.
2. Merge the relevant task branches into the integration branch.
3. Resolve conflicts by preserving the intent of all tasks.
4. Run validation again.
5. Commit the integrated result.
6. Push the integration branch.
7. Create or update an integration PR.
8. Deploy an integration preview.
9. Report:
   - task branch names
   - task PR URLs
   - task preview URLs
   - integration branch name
   - integration PR URL
   - integration preview URL
   - conflicts resolved
   - validation result
   - remaining risks

If two task branches conflict semantically, do not choose one silently. Explain the conflict and implement the safest combined behavior when possible.

---

## Production Deployment Rule

Individual task branches must not be deployed directly to production.

Production deployment is allowed only when:

1. The task PR or integration PR is merged into the production branch.
2. Validation passes.
3. The production pipeline runs from the merged branch.
4. Production deployment is explicitly allowed by the repository workflow or user instruction.

If asked to deploy production while parallel branches are still open, first check whether an integration branch is needed.

## Removal, Disablement, and Cleanup Tasks

Removal-style tasks still require the full preview-first workflow.

This applies to instructions such as:

```text
remove this
hide this
delete this
disable this
turn this off
clean this up
remove unused code
stop showing this component
```

Required outcome remains:

1. Branch.
2. Code change.
3. Validation.
4. Commit.
5. Push.
6. PR.
7. Preview deployment.
8. Final report.

Do not respond that deployment was skipped only because the task was a removal or disablement task.

---

## Safety and Secrets

Never commit secrets.

Do not print or expose:

- API keys
- tokens
- private keys
- database URLs
- production credentials
- `.env` contents
- service account files

If a deployment requires missing credentials, report the missing credential name or integration requirement without exposing any secret value.
